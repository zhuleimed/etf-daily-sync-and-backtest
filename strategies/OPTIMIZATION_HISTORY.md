## 1. 项目级 Bug 修复记录

### 1.1 SQL ORDER BY 位置错误

**严重程度：** 🔴 致命（影响所有独立年份回测）

**发现时间：** 2026-06-23，在按年份跑策略验证时发现

**代码位置：** `strategies/momentum_rotation/data.py` 的 `_load_single_etf()` 方法

**错误代码：**
```python
query = """
    SELECT date, open, high, low, close, volume
    FROM etf_daily
    WHERE symbol = ? AND date >= ?
    ORDER BY date
"""
params: list = [symbol, start_date]
if end_date:
    query += " AND date <= ?"        # ← 拼接在 ORDER BY 之后！
    params.append(end_date)

df = pd.read_sql_query(query, conn, params=params)
```

实际生成的 SQL：
```sql
-- 有 end_date 时：
SELECT ... FROM etf_daily WHERE symbol=? AND date>=? ORDER BY date AND date<=?
-- SQLite 将 "AND date<=?" 解释为 ORDER BY 的布尔表达式：
-- ORDER BY (date AND (date <= ?)) → 布尔值排序 → 结果倒序！
```

**症状：** 当使用 `--end 2024-12-31` 参数独立跑某一年回测时，数据倒序排列，
导致前几天的数据实际上是最后几天的，信号完全错误。

**验证：**
```python
# 有 end_date: 首行日期 = 2025-01-02（未来数据！）
# 无 end_date: 首行日期 = 2024-01-02（正确）
```

**修复：** 将 `ORDER BY date` 移到查询末尾（在 `if end_date` 块之后）。
```python
query = "SELECT ... WHERE symbol=? AND date>=?"
if end_date:
    query += " AND date<=?"
query += " ORDER BY date"  # 正确位置
```

**影响范围：** 所有独立年份的对比分析（2023/2024/2025/2026）在修复前都是错误的。
修复后各年份数据才可信。

**教训：** SQL 字符串拼接时，`ORDER BY` 必须始终在 `WHERE` 子句之后。
这个拼接代码从第一次提交就存在，几个月未被发现。

### 1.2 信号与交易同日执行（look-ahead bias）

**严重程度：** 🔴 致命（造成所有策略收益虚高）

**发现时间：** 2026-06-24，用户指出才意识到

**问题描述：** 回测引擎使用 `close[T]` 计算动量信号，同时也用 `close[T]` 执行交易。
这意味着在 bar T 上"看到收盘价后用这个收盘价成交"——但在真实世界中，收盘后市场已关闭。

**影响范围：** 全部 11 个策略引擎的 `_buy()` 和 `_sell()` 方法

**修复：**
- 回测引擎：`close[T]` → `open[T]`（用 `close[T-1]` 算信号，`open[T]` 执行）
- 模拟盘引擎：`close[T]` 信号 → `open[T+1]` 执行

**修复后收益变化（各策略对比见第 15 节）：**
- 不同策略受影响程度不同，最高变化达 +63%（ma_etf 从 +102→+191）
- 有些策略变好（牛市中开盘<收盘，买入价更低）
- 有些策略变差（vol_filter 从 +132→+117，卖出价更低）

**教训：** 这是量化回测中最容易犯也最容易被忽视的 bias。必须将
"信号时间"和"执行时间"严格分离。

### 1.3 模拟盘风控卖出状态丢失

**严重程度：** 🔴 模拟盘风控完全失效

**发现时间：** 2026-06-24，代码审计中发现

**代码位置：** `simulation/framework/engine.py` 风控处理段

**错误代码：**
```python
if risk_result and risk_result.triggered:
    if hold_sym and ... and state.position.shares > 0:
        trade = self.broker.sell(state, price=..., reason=risk_result.reason)
        report["action"] = "risk_sell"
        report["trade"] = trade
    state = self.state_mgr.load()    # ← 从磁盘重载旧状态！
    self.state_mgr.save(state)       # ← 把旧状态写回去
```

`broker.sell()` 修改了 `state` 的内存数据（增加了现金、清除了持仓），
但下一行的 `load()` 从磁盘读入了旧状态（卖出前的状态），
然后 `save()` 把旧状态存回磁盘——风控卖出**完全消失**了。

**修复：** 删除多余的 `load()` 调用，直接 `save()` 修改后的 state。

### 1.4 持仓热力图颜色溢出

**严重程度：** 🟡 影响报告生成

**发现时间：** 2026-06-23，18只ETF回测时触发

**错误信息：**
```
ValueError: There are 20 color bins including extensions, but ncolors = 8
```

**原因：** `matplotlib` 的 `Set2` 色图只有 8 种颜色，而 18 只 ETF 需要 18+1=19 种。
`BoundaryNorm` 创建的 bin 数量超过了色图颜色数。

**修复：** 在 `reporter.py` 中动态扩展色图颜色数：
```python
if len(symbols) + 1 > cmap.N:
    base = cmap(np.arange(cmap.N))
    repeats = (len(symbols) + 1 + cmap.N - 1) // cmap.N
    cmap = mcolors.ListedColormap(np.tile(base, (repeats, 1))[:len(symbols) + 1])
```

### 1.5 signal.py 命名冲突（循环导入）

**严重程度：** 🟡 影响特定运行环境

**发现时间：** 2026-06-23

**问题：** `strategies/momentum_rotation/signal.py` 与 Python 标准库 `signal` 同名。
当从项目根目录运行时（`sys.path` 包含当前目录），Python 导入 `signal` 时
会找到项目中的 `signal.py` 而非标准库的 `signal`，导致 pandas/pyarrow 导入链断裂。

**错误信息：**
```
AttributeError: partially initialized module 'pandas' has no attribute 'DataFrame'
```

**修复：** 重命名 `signal.py` → `momentum_signals.py`，同时更新所有 import 引用。

**涉及文件：** `momentum_rotation/engine.py`, `market_regime_rotation/engine.py`
等所有引用 `from .signal import` 的文件。

## 2. 核心交易逻辑的演进

### 2.1 时序规则的三个版本

| 版本 | 信号时间 | 执行时间 | 存在时间 | 评价 |
|------|---------|---------|---------|------|
| **v0（原始）** | `close[T]` | `close[T]` | 首次提交 ~ 2026-06-24 | ❌ look-ahead bias |
| **v1（回测修正）** | `close[T-1]` | `open[T]` | 2026-06-24 起 | ✅ 正确 |
| **v2（模拟盘 T+1）** | `close[T]` | `open[T+1]` | 2026-06-24 起 | ✅ 正确 |

### 2.2 涨跌停检查的加入

最初模拟盘没有涨跌停检查。用户指出后加入：

```python
def _check_limit_open(symbol, open_price, prev_close):
    limit_pct = 0.20 if symbol in 创业板/科创板 else 0.10
    if open_price >= prev_close * (1 + limit_pct):
        return True, "涨停"     # 不能买入
    if open_price <= prev_close * (1 - limit_pct):
        return True, "跌停"     # 不能卖出
```

**创业板/科创板 ±20%：** 159915, 159949, 588000, 588080, 588050
**普通 ETF ±10%：** 其余所有

### 2.3 回测和模拟盘为什么必须不同？

这是两类引擎的**根本目的不同**决定的：

| | 回测 | 模拟盘 |
|---|------|--------|
| 目的 | 验证策略有效性 | 模拟真实交易 |
| 数据 | 全量历史数据 | 增量每日数据 |
| 速度 | 一次性处理数百天 | 每天运行一次 |
| 状态 | 内存中 | 持久化到 JSON |
| 时序 | `close[T-1]→open[T]` | `close[T]→open[T+1]` |

两者的时序在数学上等价——信号都比执行提前约 1 个 bar。
但在具体价格上不同（`open[T]` ≠ `open[T+1]`），所以回测收益 ≠ 模拟盘收益。

原始池子只有 5 只规模指数 ETF：

**基准结果（close→open 修正前）：** +88.84%，夏普 0.75

但 2025 年宽基 ETF 普涨 16-35%，动量差距不大，策略频繁切换，2025 年仅 +6.68%（修正前）。

### 3.2 扩展至 7 只：加入创业板 + 科创板

| 新增 | 代码 | 说明 |
|------|------|------|
| 创业板ETF | 159915 | 易方达，A股成长板块代表 |
| 科创50ETF | 588000 | 华夏，科创板代表 |

**结果对比：**

| 指标 | 5只 | 7只 | 变化 |
|------|-----|-----|------|
| 全周期收益 | +88.84% | **+125.12%** | ↑ +36% |
| 2025年收益 | +6.68% | **+43.41%** | ↑ 大幅改善 |
| 夏普 | 0.75 | **1.06** | ↑ |

**为什么不是 look-ahead bias？** 创业板和科创板是 A 股市场的独立市场层次，
类似于主板、中小板，不是"2025年涨得好所以加上的行业板块"。
缺少成长板，策略就缺失了 A 股一半的市值空间。
真正有 bias 问题的是手动添加行业 ETF（半导体ETF、酒ETF），这类做法已被排除。

### 3.3 扩展至 18 只：行业 + 跨境（失败）

**候选池：** 在 7 只基础上加入红利ETF、证券ETF、军工ETF、半导体ETF、酒ETF、
医疗ETF、光伏ETF、新能源车ETF、消费ETF、纳指ETF、恒生ETF、黄金ETF

**结果：** 仅 **+65.64%**，夏普 0.76

**失败原因分析：**
1. **同质化严重：** 行业 ETF 之间高度相关（半导体涨→芯片涨），信号互相干扰
2. **交易日历冲突：** 纳指ETF、恒生ETF的交易日与 A 股不同，减少共同交易日
3. **噪声增加：** 更多标的 → 更多切换 → 更高成本（18只时交易成本 142 元 vs 7只时 97 元）
4. **信号稀释：** 动量信号在大量同质化标的中难以区分强弱

**教训：** 更多 ≠ 更好。高质量、低相关性的标的池优于大而全的池子。

### 3.4 结论：7 只是最优平衡点

| 池子大小 | 收益 | 夏普 | 切换 | 评价 |
|---------|------|------|------|------|
| 5只 | +88.84% | 0.75 | 2 | 太保守，2025年缺失成长行情 |
| **7只** | **+125.12%** | **1.06** | **19** | ✅ 完整市场覆盖 |
| 18只 | +65.64% | 0.76 | 35 | ❌ 噪声过大 |

## 5. momentum_vol_filter — 波动率过滤轮动

### 5.1 设计思路的由来

之前的过滤器（均线/绝对动量）都在价格维度上做文章，
但它们的共同问题是"价格在牛市中频繁误触发"。

波动率是**完全不同的维度**——它不判断"市场在涨还是跌"，
而是判断"市场是否有清晰的趋势"。

```
低波动 → 趋势清晰 → 动量有效 → 正常轮动
高波动 → 趋势不稳 → 动量反转 → 空仓回避
```

### 5.2 波动率阈值全测试

| 阈值 | 总收益 | 回撤 | 夏普 | 回撤持续天数 | 切换 | 评价 |
|------|--------|------|------|------------|------|------|
| 25% | +62.6% | -25.0% | 0.79 | 250天 | 20 | ❌ |
| 28% | +65.7% | -25.0% | 0.82 | 250天 | 19 | ❌ |
| **30%** | **+124.6%** | **-25.0%** | **1.19** | **155天** | **19** | ✅ |
| 32% | +124.6% | -25.0% | 1.19 | 155天 | 19 | ≈ |
| 35% | +124.6% | -25.0% | 1.19 | 155天 | 19 | ≈ |

**30% 的物理含义：** A 股年化波动率日常在 15-25% 之间。
30% 以上表示市场处于"异常高波动"状态（如 2024年1月雪球产品爆仓期间）。
只有在这个阈值下，波动率过滤才触发，且触发期间确实不适合做动量。

### 5.3 波动率窗口全测试

| 窗口 | 总收益 | 回撤 | 夏普 | 特征 |
|------|--------|------|------|------|
| 10天 | +66.3% | -25.0% | 0.82 | ❌ 太敏感 |
| **20天** | **+124.6%** | **-25.0%** | **1.19** | ✅ **选择** |
| 30天 | +125.2% | -25.0% | 1.20 | ≈ 略优但无本质差异 |
| 60天 | +130.9% | -28.6% | 1.14 | 平滑过度，错过触发时机 |

**选择 20 天的理由：**
- 30/60天结果略有不同但无本质改善
- 20天是 VaR 计算的行业标准窗口
- 较短窗口对波动率变化响应更快

### 5.4 ADJUSTMENT_DAYS 配合测试

在震荡市（如 2024上半年）中，这种反复进出会导致显著亏损。
这在一个数据周期内无法验证。

## 8. momentum_dual — 双动量（绝对+相对）

### 8.1 测试结果

| 模式 | 总收益 | 回撤 | 夏普 | 切换 |
|------|--------|------|------|------|
| **双动量** | **+64.29%** | **-29.38%** | **0.70** | **33** |
| 纯动量（基准） | +125.12% | -25.80% | 1.06 | 19 |

### 8.2 失败原因深度分析

关闭绝对动量检查反而更好——这反直觉的结果揭示了：

```
波动率过滤（成功的例子）：
  高波动 → 离场 → 低波动 → 回来
  判断的是"趋势是否清晰"，在牛熊中都只在需要时触发

绝对动量（失败的例子）：
  动量<0 → 离场 → 动量>0 → 回来
  判断的是"从特定时间点到现在是否上涨"
  20日动量在正常回调中轻易转负
  → 卖出后等动量转正 → 错过了修复性反弹
```

**一个典型场景：**
```
第1天：close=100，动量=+2%（正常回调前）
第5天：close=95，动量=-3%（正常回调）→ 卖出 ❌
第10天：close=100，动量=+1%（已修复）→ 买入，晚了两天
第20天：close=110，动量=+5%（涨完了）
```

这个场景在 2024 年反复出现，导致策略频繁被震荡出局。

### 8.3 为什么不用更长周期做绝对动量？

如果使用 60 日动量做绝对过滤（60日动量<0 才清仓）：
- 能避免小幅回调的误触发
- 但 60 日动量反应更慢，真正转负时已经跌了很多
- 在 2024-2026 牛市中几乎不会触发

**结论：** 绝对动量过滤在牛市中效果不佳，在熊市中可能有效但无法验证。
这不是一个"参数问题"，而是根本的设计问题。

**对数比价 vs 简单比价：**

| 指标 | 简单比价 | 对数比价 |
|------|---------|---------|
| 公式 | `price_a / price_b` | `log(price_a / price_b)` |
| 分布形态 | 偏态 | **正态**（z-score 的前提假设） |
| 对称性 | 非对称（A/B vs B/A 不等价） | **对称**（log(A/B) = -log(B/A)） |
| z-score 准确性 | 偏差大 | **更准确** |

**自适应阈值：**

公式：`open_threshold = ZSCORE_OPEN × (1 + (vol_spread / std_s - 1) × 0.5)`

当价差的短期波动率高于长期波动率时，阈值自动加宽（最大 1.5倍），
避免在高波动区间被噪声触发。当波动率低时，阈值收窄（最小 0.8倍），
捕捉更多交易机会。

## 12. non_momentum_strategies — 非动量策略合集

### 12.1 布林带轮动（bollinger_rotation）

**BB_STD 参数为何不影响结果？**

信号计算公式：`- (close - lower) / (upper - lower)`
其中 `upper = MA + BB_STD × std`，`lower = MA - BB_STD × std`

分子分母同时受 BB_STD 影响：
- BB_STD 变大 → upper 上移，lower 下移 → 分母 (upper-lower) 变大
- 位置百分比 (close - lower) / (upper - lower) 对 BB_STD 的变化不敏感
- 所有 ETF 的排名顺序不受 BB_STD 影响

**217 次切换的根源：** 每日计算布林带位置 → 排名变化 → 切换。
即使在窄幅震荡中，ETF 之间的排名也会每日微小波动，导致频繁切换。

**修复方向：** 需要加最小持仓天数过滤（MIN_HOLD_DAYS），或使用更长的布林带周期。

### 12.2 唐奇安通道突破（donchian_breakout）

**对价：+67.40%，108次切换**

**信号公式：**
```python
if close >= upper:   # 突破60日最高价 → 强烈买入信号
    score = close / upper (正值越大越好)
elif close <= lower: # 跌破60日最低价 → 强烈卖出信号  
    score = -(lower / close) (负值越大越好)
else:                # 在通道内
    score = (close - lower) / (upper - lower) - 0.5 # 相对位置
```

**问题：** 突破后的假突破太多。在 2024 年 6-9 月的震荡下跌中，
每次小幅反弹都触发"突破买入"，然后继续跌，反复止损。

### 12.3 量价配合（vol_price_momentum）

**信号公式：** `score = momentum × (volume / volume_ma)`

**问题：** 成交量的放大倍数极不稳定。某些日子的成交量因特殊事件（如 MSCI 调整）
异常放大，导致信号失真。

**典型错误信号：**
- 2024年9月30日：成交量突然放大 5 倍（国庆前），但第二天就恢复正常
- 策略在成交量放大日买入，次日成交量回落 → 信号消失 → 切换

**结论：** 日线级别成交量数据的噪声太大，不太适合作为辅助信号。

### 12.4 低波动率轮动（low_vol_rotation）

**逻辑：** 持有过去 N 日波动率最低的 ETF（低波动异象）

**结果：** +32.36%，夏普 0.58

**为什么差：** 低波动率异象在美股有效，但在 A 股 ETF 上不明显。
在 2024-2026 牛市中，低波动 ETF（如上证50）涨幅远低于高波动 ETF（如创业板/科创50）。
持有低波动标的意味着稳定但跑输大盘。

### 12.5 均值回归轮动（mean_reversion）

**逻辑：** 持有距 20 日均线最远的 ETF（乖离率最低 = 超跌最严重）

**结果：** +80.21%，但切换 136 次，成本 7.26%

**为什么切换这么多：** 乖离率排名每日剧烈波动。今天跌最惨的 ETF，
明天可能反弹导致乖离率快速修复 → 排名下降 → 切换到下一个"跌最惨"的 → 反复切换。

**结论：** 均值回归在趋势市中天然不适用——越跌越买的策略在牛市中会错过涨幅。
在震荡市中可能有效，但 2024-2026 不是震荡市。

## 13. market_regime_rotation — 已放弃的市场状态识别

### 13.1 设计初衷

在纯动量轮动基础上增加 BULL/BEAR 市场状态识别：
- BULL（沪深300 > 60日均线 + 20日动量 > 3%）：**锁定持有**，不做切换
- BEAR（不满足牛市条件）：正常动量轮动

### 13.2 首次结果：+37.88%

远低于纯动量的 +88.84%（close→open 修正前）。

### 13.3 问题诊断

**BULL 状态占比仅 29%，切换过于频繁。** 沪深300在60日均线附近反复穿越，
单日穿越就导致 BULL↔BEAR 状态切换。

**修复尝试：** 加入迟滞带 + 确认窗口
- 进入 BULL 需要价格 > MA × (1 + 2%)（提高门槛）
- 退出 BULL 需要价格 < MA × (1 - 1%)（降低门槛）
- 连续 5 天同向信号才切换

**修复后：** BULL占比提升到 52%，但收益仅 +39.02%

### 13.4 根本问题：逻辑反了

锁定 BULL 状态下**不切换**——这意味着在牛市中持有单一 ETF 而不切换到更强的标的。
动量轮动的核心优势就是"在趋势中不断更换到最强标的"，
锁定持有了恰恰取消了这一优势。

**正确的逻辑应该是：**
- BULL → 正常轮动（牛市中有更强的标的，应该切换）
- BEAR → 空仓或谨慎轮动（熊市中回避）

这其实和波动率过滤的思路一致。

**最终结论：** 动量轮动本身已隐含趋势跟踪能力，不需要额外的市场状态识别层。
添加 BULL/BEAR 状态机属于画蛇添足。该目录保留在仓库中作为实验记录。

## 14. 风控模式 A/B/C 对比测试

### 14.1 测试结果（momentum_rotation 上）

| 模式 | 总收益 | 回撤 | 夏普 | 切换 | 交易胜率 |
|------|--------|------|------|------|---------|
| **A（纯信号）** | **+125.12%** | **-25.80%** | **1.06** | **19** | **57.1%** |
| B（全风控） | 明显降低 | 改善有限 | 下降 | 更多 | 更低 |
| C（极端回撤） | 接近 A | 接近 A | ≈1.06 | ≈19 | ≈57% |

### 14.2 为什么风控对动量策略有害

**止损的问题：**
- 动量策略持有"趋势最强"的标的。在趋势中，价格波动是正常的。
- 5% 止损在牛市中会被正常的回调触发（如 2024年1月 ETF 普跌 5-8%）
- 一旦卖出，趋势继续上涨，你就错过了

**移动止盈的问题：**
- 动量策略在趋势中不应止盈——趋势的末端往往是最陡峭的部分
- 10% 盈利 + 5% 回撤止盈：在 2025年创业板上涨 30% 的过程中，
  多次触发止盈，每次都卖飞

**极端回撤的问题：**
- 15% 极端回撤触发清仓：等于"在最大回撤发生后卖出"
- 这时往往已经是底部了

### 14.3 波动率过滤 vs 传统止损

这是本项目最重要的发现之一：

```
传统止损（价格维度）：价格跌了 → 卖出
                   → 在牛市中频繁误触发
                   → 打断趋势，降低收益

波动率过滤（波动维度）：市场高波动 → 不交易
                    → 只在真正混乱时才触发
                    → 不影响正常趋势跟踪
```

波动率过滤本质上也是一种风控，但它是在**信号层面**（不产生交易信号）
而非**执行层面**（强制平仓），所以不会打断趋势。

## 15. close→open 价格修正详细影响

### 15.1 修正方法与影响

| 策略 | 修正前(close) | 修正后(open) | 变化 | 变化原因 |
|------|-------------|-------------|------|---------|
| rotation | +125.12% | **+132.93%** | **↑7.81%** | 买入价降低 |
| vol_filter | +131.88% | **+116.59%** | **↓15.29%** | 卖出价降低 > 买入价降低 |
| ma_filter | +124.80% | **+166.93%** | **↑42.13%** | MA突破后开盘买入价更低 |
| ma_etf | +102.02% | **+191.0%** | **↑88.98%** | 同上，放大效应更强 |
| dual | +64.29% | **+86.3%** | **↑22.01%** | 买入价更低 |

### 15.2 为什么有的涨有的跌

**涨的策略（rotation, ma_filter, ma_etf, dual）：**
这些策略的买入信号发生时，`open[T]` 通常低于 `close[T-1]`
（隔夜回调），因此买入价更低 → 持仓成本更低 → 收益更高。

**跌的策略（vol_filter）：**
波动率过滤策略的触发逻辑不同——在高波动时卖出。
高波动期间的隔夜跳空往往是向下的，`open[T]` 低于前一天收盘，
卖出价更低 → 收益受损。

### 15.3 修正后的策略排名变化

**修正前排名（用 close 执行）：**
1. vol_filter +131.88% ⬅ 最优
2. rotation +125.12%
3. ma_filter +124.80%
4. ma_etf +102.02%
5. dual +64.29%

**修正后排名（用 open 执行）：**
1. **ma_etf +191.0%** ⬅ 新第一
2. **ma_filter +166.93%** ⬅ 第二
3. rotation +132.93%
4. vol_filter +116.59%
5. dual +86.3%

ma_etf 跃升第一，但这**不代表它比 vol_filter 更好**——ma_etf 的高收益
部分来自"隔夜跳空"的有利影响，这在特定市场环境中可能会反转。

**推荐的策略选择标准：** 夏普比率 > 收益排名。vol_filter 的夏普 1.31 仍是最高的。

## 16. SQL ORDER BY Bug 全过程

### 16.1 发现过程

在跑 2024/2025/2026 各年份独立回测时，发现 2024 年独立跑的结果与
全周期中 2024 年片段的收益不一致：

```
全周期中的 2024 年: +18.76%
独立跑 2024:          -9.38%  ← 明显错误
```

### 16.2 排查过程

1. **参数一致性检查：** 全周期和独立跑的 config 参数一致 → 排除
2. **数据加载检查：** SQLite 筛选测试 `date <= '2024-12-31'` 正确 → 排除
3. **逐行检查数据：** 发现第一条数据日期是 `2025-01-02` 而非 `2024-01-02`
   → 数据倒序！
4. **原因定位：** `_load_single_etf` 的 SQL 拼接 → `ORDER BY date` 在 `WHERE` 子句之前

### 16.3 修复验证

修复后各年份独立跑与全周期结果一致：

```
2024 全周期: +18.76%  |  独立跑: +18.76%  ✅ 一致
```

## 17. 模拟盘框架构建记录

### 17.1 框架设计

T+1 待执行订单模式的完整状态机：

```
无仓位 + 无待执行订单
  →  信号触发 → 创建 pending_order（"明天开盘买X"）
  →  次日开盘 → 执行订单（检查涨/跌停）
  →  已有仓位

已有仓位（持有X）
  →  信号触发切换 → 创建 pending_order（"明天卖X买Y"）
  →  次日开盘 → 检查涨跌停：
      →  X跌停 → 卖不出，取消整个切换
      →  Y涨停 → 买不进，取消整个切换
      →  都没问题 → 执行切换
  →  持有新仓位
```

### 17.2 已实现的保护措施

| 措施 | 实现位置 | 说明 |
|------|---------|------|
| 涨跌停检查 | engine._execute_pending_order | 开盘价涨/跌停则取消订单 |
| today_opened 保护 | broker.buy → state.position.today_opened | 当天买入不能当天卖 |
| 数据到位检查 | daily.py → latest_day == today_str | 数据未同步完不交易 |
| 状态原子写入 | state.py → tempfile + os.replace | 防文件损坏 |
| 最小持仓天数 | engine → days_since_switch < MIN_HOLD_DAYS | 防噪声切换 |

### 17.3 已知局限

1. **未处理分红/送股：** ETF 分红会影响持仓成本，当前未处理
2. **未处理停牌：** ETF 停牌时无法交易，当前无此判断
3. **涨跌停使用开盘价判断：** 盘中打开涨跌停的情况未考虑（保守估计）

## 18. 管线编排器构建记录

### 18.1 架构变迁

**v0（简单 cron）：** 数据同步和模拟盘各自独立 cron，互不关联。
问题：模拟盘可能在数据同步完成前运行。

**v1（pipeline 顺序执行）：** 一个 pipeline.py 统一调度：
1. 先执行数据同步（`main.py --sync-only`）
2. 同步成功后执行模拟盘

**优势：** 不依赖固定时间，同步花多久模拟盘就等多久。

### 18.2 交易日判断

```python
def is_trade_day(check_date):
    if check_date.weekday() >= 5:       # 周末过滤
        return False
    from chinese_calendar import is_workday
    return is_workday(check_date)       # 法定节假日（春节/国庆等）
```

`chinese_calendar` 库内置了 A 股交易日数据，准确率较高。

### 18.3 子进程日志

最初用 `subprocess.run()` ＋ `capture_output=True`，子进程的输出被捕获到变量中
但不打印到主进程日志。出问题时无法看到子进程的输出。

**改为 `subprocess.Popen()` + 实时流式打印：**
```python
proc = subprocess.Popen([python] + cmd, stdout=PIPE, stderr=PIPE, text=True)
for line in iter(proc.stdout.readline, ""):
    print(line)                      # 实时打印
    stdout_lines.append(line)       # 同时保存
```

### 18.4 状态文件自我修复

`PipelineStatus.needs_rerun()` 检测上次运行状态：
- `running`：上次异常中断 → 重置重新运行
- `failed`：上次运行失败 → 重新运行
- `completed` / `skipped`：正常状态 → 不重复运行

## 19. 技术指标尝试（ADX/RSI）

### 19.1 ADX 尝试（已放弃）

**思路：** ADX > 25 表示强趋势 → 动量轮动；ADX < 25 表示弱趋势 → 空仓

**实现：** 在 `market_regime_rotation` 中尝试加入 ADX 指标

**结果：** 不成功。ADX 在 A 股 ETF 上的信号不稳定，
与动量信号冲突过多，导致切换次数增加但收益没改善。

### 19.2 RSI 尝试（已放弃）

**思路：** RSI > 70 超买 → 卖出；RSI < 30 超卖 → 买入

**结果：** 在动量策略背景下 RSI 信号与动量信号高度重叠。
RSI 超买时动量往往已经很高，卖出会提前下车。
RSI 超卖时动量往往已经转负，买入不符合动量逻辑。

**结论：** 传统技术指标（ADX/RSI）与动量轮动策略的兼容性不佳，
且在这套框架中实现复杂、收益改善有限。

## 20. 数据同步模块历史问题

### 20.1 Tencent → Sina 双轨制

**切换逻辑：**
1. 默认使用 Tencent 接口（`web.ifzq.gtimg.cn`）
2. Tencent 连续失败 → 自动切换到 Sina
3. 每 50 次 Sina 请求尝试恢复 Tencent
4. 日志标记当前活跃源

**已知问题：**
- Tencent 在大盘剧烈波动日（如 2024年9月底暴涨）可能限流
- Sina 的指数数据偶尔缺失
- 双轨制解决了大部分稳定性问题

### 20.2 交易日判断问题

`chinese_calendar` 的 `is_workday()` 对于调休工作日（如春节前一个周六上班）
返回 `True`，但 A 股在这些调休日是不交易的（只有银行间市场交易）。

**修复：** 使用 `is_trade_day()` 判断：周末过滤 + chinese_calendar + 调休日过滤。

### 20.3 etf_sync 与 004_sequoia-x 的隔离

本项目不从 004_sequoia-x 导入任何模块。唯一共用的是 WxPusher 的 Token
（从 `.env` 读取），这是配置而非依赖。即使 004 项目停止运行，
本项目也不受影响。

## 21. 策略开发踩坑记录

### 21.1 常见错误汇总

| 错误 | 现象 | 原因 | 预防 |
|------|------|------|------|
| look-ahead bias | 回测收益虚高 | 用 close[T] 同时算信号和执行 | 使用 signal_idx = max(1, idx-1) |
| SQL 顺序错误 | 独立年份回测结果错误 | ORDER BY 位置错误 | 检查 SQL 拼接 |
| 风控状态丢失 | 模拟盘风控不生效 | load() 覆盖了 sell() 的修改 | Review 代码 |
| 色图溢出 | 报告生成失败 | ETF 数量 > 色图颜色数 | 动态扩展 |
| 信号函数命名冲突 | 循环导入 | signal.py 与 stdlib 同名 | 避免标准库同名 |
| 正则替换破坏代码 | 文件语法错误 | regex 匹配了意料之外的代码 | 逐行 diff 验证 |

### 21.2 参数优化的陷阱

1. **过拟合：** 在同一个数据集上调参过多 → 结果不可推广
   - 预防：先确定"合理的参数范围"，在此范围内寻找稳定区间而非单一点
   - 实例：vol_filter 的 30% 阈值在 28%-35% 范围内都是稳定的

2. **未来信息泄漏：** 使用全局数据（如全周期均值）而非滚动计算
   - 预防：始终使用 `shift()` 或 `rolling()` 计算

3. **数据透视偏差：** 根据某个特定年份的表现选择参数
   - 预防：分年验证，确保各年份表现合理
   - 实例：pair_trading 的最后三对选择在 2024/2025/2026 都经过验证

### 21.3 开发顺序

正确的策略开发顺序：
```
1. 基础回测（默认参数）
2. 检查是否有 look-ahead bias
3. 参数敏感度测试（单变量）
4. 交互参数测试（双变量）
5. 分年验证
6. 等权基准对比
7. 模拟盘适配
8. pipeline 集成
```
